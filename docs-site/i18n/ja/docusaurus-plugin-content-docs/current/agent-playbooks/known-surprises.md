# 既知の驚き

このファイルは、エージェントの間違いを引き起こしたリポジトリ固有の混乱点を追跡します。

## エントリ基準

すべてが当てはまる場合にのみエントリを追加します。

- このリポジトリに固有のものです（一般的なアドバイスではありません）。
- 将来のエージェントでも再発する可能性があります。
- これには、次のような具体的な軽減策があります。

不明な場合は、エントリを追加する前に開発者に問い合わせてください。

## エントリ テンプレート

```md
### [Short title]

- **Date:** YYYY-MM-DD
- **Observed by:** agent name or contributor
- **Context:** where/when it happened
- **What was surprising:** concrete unexpected behavior
- **Impact:** what went wrong or could go wrong
- **Mitigation:** exact step future agents should take
- **Status:** confirmed | superseded
```

## エントリ

### ポートレスにより正規のローカル アプリ URL が変更されます

- **日付:** 2026-03-18
- **観察者:** Codex
- **コンテキスト:** ブラウザの検証とスモーク フロー
- **驚いたこと:** デフォルトのローカル URL は通常の Vite ポートではありません。リポジトリはポートレス経由の `http://bitsocial.localhost:1355` を想定しているため、`localhost:3000` または `localhost:5173` をチェックすると、間違ったアプリがヒットするか、まったくヒットしない可能性があります。
- **影響:** 開発サーバーが正常な場合でも、ブラウザ チェックが失敗したり、間違ったターゲットを検証したりする可能性があります。
- **軽減策:**まずは`http://bitsocial.localhost:1355`。明示的に直接 Vite ポートが必要な場合にのみ、`PORTLESS=0 corepack yarn start` でバイパスしてください。
- **ステータス:** 確認済み

### コミットフックが非対話型コミットをブロック

- **日付:** 2026-03-18
- **観察者:** Codex
- **コンテキスト:** エージェント主導のコミット ワークフロー
- **驚いたこと:** `git commit` トリガーHusky を介してコミットし、対話型 TTY 入力を待機します。これにより、非対話型エージェント シェルがハングします。
- **影響:** 通常のコミット中にエージェントが無期限に停止する可能性があります。
- **軽減策:** エージェントが作成したコミットには `git commit --no-verify -m "message"` を使用します。人間は引き続き `corepack yarn commit` または `corepack yarn exec cz` を使用できます。
- **ステータス:** 確認済み

### Yarn クラシックを回避するにはコアパックが必要です

- **日付:** 2026-03-19
- **観察者:** Codex
- **内容:** パッケージ マネージャーの Yarn 4 への移行
- **驚いたこと:** マシンにはまだグローバルYarn クラシックは `PATH` にインストールされるため、プレーン `yarn` を実行すると、固定された Yarn 4 バージョンではなく v1 に解決される可能性があります。
- **影響:** 開発者がリポジトリのパッケージ マネージャーの固定を誤ってバイパスし、異なるインストール動作またはロックファイル出力が得られる可能性があります。
- **軽減策:**シェル コマンドの場合は `corepack yarn ...`、または最初に `corepack enable` を実行してプレーン `yarn` が固定された Yarn 4 バージョンに解決されます。
- **ステータス:** 確認済み

### ポートレス アプリ名が Bitsocial Web ワークツリー間で衝突する問題を修正しました

- **日付:** 2026-03-30
- **観察者:** Codex
- **コンテキスト:** ある Bitsocial Web ワークツリーで `yarn start` を開始する一方で、別のワークツリーが既にポートレス経由で提供されている
- **驚いたこと:** すべてのワークツリーでリテラルのポートレス アプリ名 `bitsocial` を使用すると、バッキング ポートが異なる場合でもルート自体が衝突するため、2 番目のプロセスが`bitsocial.localhost` がすでに登録されているため失敗します。
- **影響:** ポートレスが安全に共存できるようにすることを目的としている場合でも、並列 Bitsocial Web ブランチは相互にブロックする可能性があります。
- **緩和策:** ポートレスの起動を `scripts/start-dev.mjs` の背後に保持し、正規のケース外のブランチ スコープの `*.bitsocial.localhost:1355` ルートを使用するようになりました。ベア `bitsocial.localhost` 名がすでに占有されている場合は、ブランチ スコープのルートにフォールバックします。
- **ステータス:** 確認済み

### ポート 3001 をハードコーディングするためにドキュメント プレビューが使用されます

- **日付:** 2026-03-30
- **観察者:** Codex
- **コンテキスト:** `yarn start` を他のローカル リポジトリおよびエージェントと並行して実行します
- **驚くべき点:** root dev コマンドは、`docusaurus start --port 3001` を使用してドキュメント ワークスペースを実行しました。そのため、メイン アプリがすでに Portless を使用していても、別のプロセスがすでに `3001` を所有している場合は常に、開発セッション全体が失敗しました。
- **影響:** `yarn start` は、起動直後に Web プロセスを強制終了し、ドキュメント ポートを介した無関係なローカル作業を中断する可能性がありました。 collision.
- **軽減策:** ドキュメントの起動を `yarn start:docs` の背後で維持します。これにより、ポートレスと `scripts/start-docs.mjs` が使用され、挿入された空きポートが優先されるか、直接実行時に次の利用可能なポートにフォールバックされます。
- **ステータス:** 確認済み
