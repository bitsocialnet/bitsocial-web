# 去中心化 Twitter/X

第 3 阶段计划打造一个聚焦、去中心化的 Twitter/X 替代方案。其核心是文本优先的公共对话：短帖、回复、转帖、关注、实时讨论和社区，同时把平台层开放出来。

Twitter/X 至今仍由帖子、文本和思想分享所定义。第 3 阶段客户端应该围绕这一核心体验竞争，并把它做到极致。

本页描述的是产品方向，而不是锁定的发布规格。随着协议和早期应用逐渐成熟，具体界面、默认信息流、广告模式、AI 功能和 RPC 市场都可以变化。

## 应该证明什么

该客户端应该证明，基于个人资料的社交网络可以避免成为托管平台：

- 用户可以拥有自己的身份和个人资料
- 社区和个人资料节点可以保持点对点
- 社区可以在 Bitsocial 客户端之间产生网络效应
- RPC提供商可以让应用程序方便而无需托管
- Feed 算法可以是可选服务，而不是平台法
- 其他客户端仍然可以竞争同一网络

目的是打造尽可能强大的公共对话客户端，并展示协议能够延伸多远。

## 目的熟悉，设计可替换

默认体验应该能与 Twitter/X 的核心竞争：快速的首页信息流、文本帖子、关注、回复、转帖式传播、社区、通知、搜索，以及开箱即用的 For You 排名视图。

Bitsocial Forge 可以运行第一个默认的 RPC 和信息流服务。这个默认设置可以包含排名信息流和广告，让应用从第一天起就感觉完整，而不是要求主流用户自己组装整套服务。

不同之处在于，默认选项不应变成牢笼。用户应该能够切换 RPC、信息流、实例、排名系统、广告和内容发现提供商，或彻底移除排名。客户端可以在首次启动时提供明确的默认选择，同时让每项主要服务都保持可替换。

这使客户端比传统平台更可定制。一位用户可以保留带广告的默认排名信息流，另一位可以使用不排名的时间顺序信息流，还有人可以选择注重隐私的 RPC、由社区运营的内容发现服务、付费无广告信息流，或为特定亚文化构建的小众算法。

## 跨客户端社区

社区应该比一个应用程序内的孤立群体重要得多。

在 X/Twitter 上，社区被限制在 X 内部。它们可以有用，但继承了单一平台、单一账户系统、单一推荐栈和单一产品界面的限制。

Bitsocial 社区可以通过不同的客户端创建、托管、发现和使用。这意味着第 3 阶段客户端可以显示来自更广泛 Bitsocial 网络的社区和帖子，而不仅限于从该客户端开始使用网络的用户。一个社区可以同时汇集来自 imageboard 客户端、Reddit 式讨论客户端、小众论坛客户端、移动应用和第 3 阶段客户端的活动。

这就是核心的网络效应优势：一个客户端可以让主流用户感到熟悉，同时仍然从许多客户端、社区节点、RPC 提供商和独立服务中获得价值。

## 可选的馈送算法

第 3 阶段客户端不应把单一的全球排名系统强加给所有人。

信息流算法应该是选择加入的。用户可以从市场中选择算法、切换提供商、使用公司提供的算法、使用匿名运营者运行的算法、使用社区构建的算法、运行个人算法，或者完全不使用算法。

公共 RPC 提供商是这些服务竞争的天然场所。他们可以对内容进行索引、排名和推荐，但他们不应该拥有用户或个人资料。

这些服务还可以在应用程序本身的形状上进行竞争。一个 RPC 可能会提供带有广告的排名提要。另一个可能会提供未排名的按时间顺序排列的提要。另一个可能专注于隐私、翻译、审核、社区发现或利基社交图谱。

如果经济模式成立，由 RPC 支撑的信息流服务可以加入类似主流平台正在尝试放进信息流的 AI 功能：自动翻译、摘要、机器人辅助回复、搜索答案、审核辅助，或社区注释式上下文。

这些功能应该是服务选择，而不是协议要求。默认 RPC 可以通过提供更丰富的信息流来竞争，但用户和竞争客户端仍应能够选择更简单、私密、按时间排序、无广告或特定社区的替代方案。

## 非托管 RPC

每个用户都应该能够通过 RPC 作为完整的点对点节点参与，而无需让 RPC 提供商拥有其身份或个人资料。

托管路径很重要，因为大多数用户不会通过运行服务器来启动。退出路径同样重要：用户应该可以在任何时候把自己的个人资料节点迁移到低规格硬件上，包括 Raspberry Pi。

这就是便利和托管之间的区别。

## Bitsocial Chain 增强的公共对话

Bitsocial Chain 可以把持久命名、支付、打赏、奖励和其他金融基础设施直接带入公共对话。

第 3 阶段客户端继续以帖子、文本、思想分享和实时讨论为中心，同时与其他 Bitsocial 客户端共享社区和网络效应。
