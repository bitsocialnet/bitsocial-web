# परमिशनलेस सार्वजनिक RPC

मूळ सार्वजनिक RPC प्रस्ताव जुन्या plebbit शब्दावलीत लिहिलेल्या GitHub समस्या म्हणून जगला. हे पृष्ठ त्या कल्पनेला बिटसोशल भाषेत पुनर्लेखन करते आणि अंमलबजावणी तपशीलाच्या भिंतीऐवजी उत्पादन-स्तरीय प्रस्ताव म्हणून फ्रेम करते.

## साध्या-भाषेचे ध्येय

Bitsocial Forge सार्वजनिक RPC सेवा चालवू शकते जी अनेक वापरकर्त्यांना त्यांचे स्वतःचे बिटसोशल समुदाय दूरस्थपणे व्यवस्थापित करू देते, ऑपरेटरमध्ये बदल न करता. समुदाय.

सेवेने मोबाइल आणि लाइटवेट क्लायंटला तीन मर्यादा जपून व्यावहारिक बनवायला हवे:

1. वापरकर्ते डीफॉल्टनुसार एकमेकांपासून अलिप्त राहतात.
2. परवानग्या स्पष्ट आणि बारीक राहतात.
3. सध्याच्या दरम्यान RPC सह सुसंगतता आणि प्रतिसादाची विनंती केली जाऊ शकते. रोलआउट.

## कोणती समस्या सोडवते

आज, सर्वात सोपा RPC मॉडेल सहसा सर्व-किंवा-काहीही नसतो: एक प्रमाणीकरण की, एक प्राधिकरण डोमेन, पूर्ण प्रवेश. ते एकाच ऑपरेटरसाठी कार्य करते परंतु सार्वजनिक बहु-वापरकर्ता सेवेसाठी नाही.

परवानगी नसलेल्या सार्वजनिक RPC ला सशक्त मॉडेलची आवश्यकता आहे:

- एक सेवा अनेक वापरकर्ते होस्ट करू शकते
- प्रत्येक वापरकर्त्याला त्यांचे स्वतःचे समुदाय आणि मर्यादा मिळतात
- ऑपरेटर-परिभाषित धोरणे गैरवर्तन रोखू शकतात
- वापरकर्ता तरीही दूर जाऊ शकतो किंवा नंतर स्वत: होस्ट करू शकतो

## मॉडेल

### वापरकर्ते

प्रत्येक वापरकर्त्याला प्रमाणीकरण क्रेडेन्शियल आणि परवानगी बंडल मिळते.

### समुदाय

सेवेद्वारे तयार केलेले समुदाय मालकाच्या रेकॉर्डला नियुक्त केले जातात. मालकी स्पष्टपणे ट्रॅक केली जाते जेणेकरून व्यवस्थापन पद्धती योग्य वापरकर्त्यापर्यंत पोहोचवता येतील.

### परवानग्या

परवानग्या क्षमता-आधारित आहेत. "RPC वापरू शकतो" साठी एका बुलियनऐवजी, सर्व्हर नियंत्रित करू शकतो:

- वापरकर्ता किती समुदाय तयार करू शकतो
- कोणत्या व्यवस्थापन पद्धती उपलब्ध आहेत
- कोणत्या प्रकाशन ऑपरेशनला परवानगी आहे
- कोणत्या दर मर्यादा लागू
- कोणते प्रशासकीय पृष्ठभाग दृश्यमान आहेत

### प्रशासकीय पृष्ठभाग

सार्वजनिक RPC ने स्वतः वापरकर्त्याच्या RPC वर्तनावर लक्ष केंद्रित केले पाहिजे. वापरकर्ता निर्मिती, मालकी हस्तांतरण आणि ऑडिट पुनरावलोकन यासारखी प्रशासकीय कार्ये वेगळ्या ऑपरेटर API आणि डॅशबोर्डमध्ये आहेत.

## सुसंगतता स्थिती

वापरकर्ता-फेसिंग डॉक्युमेंटेशन **समुदाय** आणि **प्रोफाइल** सारख्या बिटसोशल संज्ञा वापरल्या पाहिजेत.

वायर स्तरावर, पहिले रोलआउट अजूनही वर्तमान JSON-RPC वाहतूक आणि पेलोड आकार जतन करू शकते जेथे ते सुसंगततेसाठी उपयुक्त आहे. दुसऱ्या शब्दात: डॉक्सला जुन्या plebbit दस्तऐवजांप्रमाणे बोलण्याची आवश्यकता नाही, जरी संक्रमण कालावधी काही लेगसी पद्धतीची नावे किंवा विनंती आकार पडद्यामागे ठेवत असेल.

## प्रस्तावित परवानगी बंडल

```ts
type PermissionBundle = {
  maxCommunities: number; // 0 = unlimited
  methods: {
    createCommunity: boolean;
    startCommunity: boolean;
    stopCommunity: boolean;
    editCommunity: boolean;
    deleteCommunity: boolean;
    publishComment: boolean;
    publishVote: boolean;
    publishCommentEdit: boolean;
    publishCommentModeration: boolean;
    publishCommunityEdit: boolean;
    getComment: boolean;
    getCommentPage: boolean;
    getCommunityPage: boolean;
    fetchContent: boolean;
    resolveAuthorAddress: boolean;
    commentUpdateSubscribe: boolean;
    communityUpdateSubscribe: boolean;
    communityListSubscribe: boolean;
    settingsSubscribe: boolean;
  };
  rateLimits: {
    requestsPerMinute: number;
    publishesPerHour: number;
  };
  storage: {
    maxTotalSize: number;
  };
  scope: {
    canPublishExternal: boolean;
    canReadExternal: boolean;
  };
  admin: {
    canTransferOwnership: boolean;
    canManageUsers: boolean;
    canViewAuditLogs: boolean;
    canViewAllCommunities: boolean;
  };
};
```

अचूक पद्धतीची नावे उदाहरणात्मक आहेत. महत्त्वाचा भाग म्हणजे धोरणाचा आकार: वैयक्तिक क्षमता एका सुपरयुझर टोकनमध्ये एकत्रित करण्याऐवजी स्वतंत्रपणे नियंत्रित केल्या जातात.

## कनेक्शन प्रवाह

```text
client connects with auth credential
-> server validates the credential
-> server loads the user's permission bundle
-> server returns a permissions notification
-> client proceeds with the subset of actions it is allowed to use
```

परवानगी जागरूकता वैकल्पिक राहिली पाहिजे. नोटिफिकेशनकडे दुर्लक्ष करणारा क्लायंट सर्व्हरवरून प्रमाणित ऑथरायझेशन अयशस्वी हाताळून तरीही योग्य रीतीने वागू शकतो.

## मालकीची अंमलबजावणी

जेव्हा सेवा समुदाय तयार करते, तेव्हा ती कॉलिंग वापरकर्त्याला आपोआप मालकी नियुक्त करते. तेथून:

- समुदाय सुरू करणे, थांबवणे, संपादित करणे आणि हटवणे या क्रिया मालकाच्या व्याप्तीत आहेत
- सूची आणि सदस्यत्व प्रतिसाद कॉलरच्या स्वतःच्या समुदायांसाठी डीफॉल्ट आहेत
- विस्तृत दृश्यमानता ही स्पष्ट प्रशासकीय परवानगी आहे, डीफॉल्ट नाही

एक धार केस खूप महत्त्वाचे आहे: जर वापरकर्त्याने समुदायाचे सदस्यत्व घेतले तर ते ** मालकीचे नाहीत**, सर्व्हरने केवळ सार्वजनिक स्थिती उघड करणे आवश्यक आहे जी कोणत्याही बाहेरील निरीक्षकाने पाहिली पाहिजे. केवळ-मालक कॉन्फिगरेशन किंवा अंतर्गत रनटाइम डेटा सदस्यता API द्वारे कधीही लीक होऊ नये.

## सुचविलेले ऑपरेटर पृष्ठभाग

प्रशासक API कंटाळवाणे आणि स्पष्ट राहू शकते:

- वापरकर्त्यांची यादी करा
- एका वापरकर्त्याची तपासणी करा
- वापरकर्ते तयार करा किंवा अपडेट करा
- वापरकर्ते हटवा
- समुदाय मालकी हस्तांतरित करा
- ऑडिट लॉगची तपासणी करा

या ऑपरेटर API साठी प्रमाणीकरण अंतिम-वापरकर्ता RPC प्रमाणीकरणापासून पूर्णपणे वेगळे असावे.

## रोलआउट फेज

### टप्पा 1

- सार्वजनिक RPC प्रकल्प संरचना स्थापित करा
- वापरकर्ता रेकॉर्ड आणि मालकी ट्रॅकिंग जोडा
- सध्याच्या RPC सर्व्हरला फाटा द्या किंवा वाढवा

### फेज 2

- परवानगी बंडल लागू करा
- RPC मेथड लेयरवर त्यांची अंमलबजावणी करा
- कनेक्टवर परवानग्या मेटाडेटा परत करा

### फेज 3

- ऑपरेटर API जोडा
- ऑडिट लॉगिंग जोडा
- प्रशासक प्रमाणीकरण जोडा

### फेज 4

- प्रशासक डॅशबोर्ड पाठवा
- चाचणी गैरवर्तन नियंत्रणे
- दर मर्यादा आणि स्टोरेज कोटा कडक करा

## उघडलेले प्रश्न

### ऑथ क्रेडेंशियल स्पॅम

सार्वजनिक सेवा तयार करण्याआधी स्वस्त स्तर तयार करणे आवश्यक असल्यास, auth चे आव्हान असू शकते क्रेडेन्शियल एक संभाव्य मार्ग म्हणजे सामुदायिक आव्हान मॉडेलचाच पुनर्वापर करणे त्यामुळे क्रेडेन्शियल जारी करणे हे उर्वरित नेटवर्क प्रमाणेच गैरवापरविरोधी तत्वज्ञानाचा वारसा घेते.

### लेगसी नामकरण

काही सुरुवातीच्या अंमलबजावणीमुळे अजूनही सुसंगततेसाठी लेगसी पद्धतीची नावे उघड होऊ शकतात. ते स्थलांतर तपशील म्हणून मानले पाहिजे, Bitsocial डॉक्सचे कायमस्वरूपी सार्वजनिक शब्दसंग्रह म्हणून नाही.

## सारांश

हा प्रस्ताव खरोखर एका गोष्टीबद्दल आहे: सार्वजनिक RPC पायाभूत सुविधांना कस्टोडिअल न बनवता उपयुक्त बनवणे. चांगल्या सार्वजनिक बिटसोशियल आरपीसीला समुदाय चालवण्यासाठी पर्यायी सहाय्य वाटले पाहिजे, नवीन केंद्रीय प्लॅटफॉर्मसारखे नाही जे मागील दाराने मालकी पुन्हा दावा करते.
