# Twitter/X を分散化する

フェーズ3では、Twitter/X に代わる、焦点を絞った分散型サービスを構築します。中心となるのはテキスト主体の公開対話です。短い投稿、返信、リポスト、フォロー、リアルタイムの議論、コミュニティを提供しながら、プラットフォーム層を開放します。

Twitter/X は今も、投稿、テキスト、アイデアの共有によって成り立っています。フェーズ3のクライアントは、この中核体験で競い、それを卓越したものにするべきです。

このページは固定されたリリース仕様ではなく、製品の方向性を説明します。正確なインターフェース、デフォルトフィード、広告モデル、AI 機能、RPC マーケットプレイスは、プロトコルと初期アプリの成熟に伴って変わる可能性があります。

## 何を証明すべきか

このクライアントは、プロフィール型ソーシャルネットワークがカストディ型プラットフォームにならずに済むことを証明すべきです。

- ユーザーはアイデンティティとプロファイルを所有できる
- コミュニティとプロファイル ノードはピアツーピアを維持できる
- コミュニティは Bitsocial クライアント全体にネットワーク効果をもたらすことができます
- RPC プロバイダーは、管理を行わずにアプリを便利にすることができます
- フィード アルゴリズムはプラットフォーム法の代わりにオプションのサービスになる可能性がある
- 他のクライアントは引き続き同じネットワークをめぐって競合することができます

目的は、可能な限り優れた公開対話クライアントを構築し、プロトコルがどこまで広がれるかを示すことです。

## 目的は親しみやすく、設計は交換可能

デフォルト体験は、Twitter/X の中核で競争力を持つべきです。高速なホームフィード、テキスト投稿、フォロー、返信、リポスト型の拡散、コミュニティ、通知、検索、そしてすぐに使えるランク付けされた For You ビューを備えます。

Bitsocial Forge は、最初のデフォルト RPC およびフィード サービスを実行できます。このデフォルトには、ランク付けされたフィードと広告を含めることができるため、メインストリーム ユーザーにスタック全体を自分で組み立てるように依頼するのではなく、初日からアプリが完成したと感じることができます。

違いは、デフォルトが牢獄になってはならないことです。ユーザーは RPC、フィード、インスタンス、ランキングシステム、広告、発見プロバイダーを切り替えたり、ランキングを完全に外したりできるべきです。クライアントは、すべての主要サービスを交換可能に保ちながら、初回起動時には明確な方針を示せます。

そのため、このクライアントは従来のプラットフォームよりも柔軟にカスタマイズできます。あるユーザーは広告付きのデフォルト順位フィードを使い続け、別のユーザーは順位付けのない時系列フィードを使えます。さらに、プライバシー重視の RPC、コミュニティ運営の発見サービス、有料の広告なしフィード、特定のサブカルチャー向けのニッチなアルゴリズムを選ぶこともできます。

## クライアントを超えたコミュニティ

コミュニティは、1 つのアプリ内の孤立したグループよりもはるかに重要である必要があります。

X/Twitter では、コミュニティは X 内に限定されます。コミュニティは便利ですが、1 つのプラットフォーム、1 つのアカウント システム、1 つの推奨スタック、および 1 つの製品サーフェスの制限を受け継ぎます。

Bitsocial コミュニティは、異なるクライアントから作成、ホスト、発見、利用できます。つまりフェーズ3のクライアントは、その中で利用を始めたユーザーだけでなく、より広い Bitsocial ネットワークのコミュニティや投稿を表示できます。1つのコミュニティに、イメージボードクライアント、Reddit 型の議論クライアント、ニッチなフォーラムクライアント、モバイルアプリ、フェーズ3のクライアントから同時に活動が集まることもあります。

これがネットワーク効果の中核的な利点です。1 つのクライアントが主流のユーザーにとって親しみを感じながらも、多くのクライアント、コミュニティ ノード、RPC プロバイダー、および独立したサービスから価値を引き出すことができます。

## オプションのフィードアルゴリズム

フェーズ3のクライアントは、全員に単一のグローバルランキングシステムを強制すべきではありません。

フィードアルゴリズムはオプトインである必要があります。ユーザーは、マーケットプレイスからアルゴリズムを選択することも、プロバイダーを切り替えることも、企業のアルゴリズムを使用することも、匿名のオペレーターが実行するアルゴリズムを使用することも、コミュニティによって構築されたアルゴリズムを使用することも、個人的なアルゴリズムを実行することも、あるいはアルゴリズムをまったく使用しないこともできます。

パブリック RPC プロバイダーは、これらのサービスが競合するのが自然な場所です。コンテンツのインデックス付け、ランク付け、推奨はできますが、ユーザーやプロフィールを所有すべきではありません。

これらのサービスは、アプリ自体の形状でも競合する可能性があります。 1 つの RPC は、広告を含むランク付けされたフィードを提供する場合があります。また、ランク付けされていない時系列フィードを提供する場合もあります。プライバシー、翻訳、モデレーション、コミュニティ発見、またはニッチなソーシャル グラフを専門とする人もいます。

経済性がうまくいけば、RPC 支援のフィード サービスは、自動翻訳、要約、ボット支援による返信、検索回答、モデレーション支援、コミュニティ ノート スタイルのコンテキストなど、主流プラットフォームがフィードに導入しようとしているものと同様の AI 機能を追加できる可能性があります。

これらの機能は、プロトコル要件ではなく、サービスの選択である必要があります。デフォルトの RPC は、より豊富なフィードを提供することで競合できますが、ユーザーと競合するクライアントは、よりシンプル、プライベート、時系列、広告なし、またはコミュニティ固有の代替を選択できる必要があります。

## 非保管 RPC

各ユーザーは、自分の ID やプロファイルに対する所有権を RPC プロバイダーに与えることなく、RPC を通じて完全なピアツーピア ノードとして参加できる必要があります。

ほとんどのユーザーはサーバーを実行することから始めないため、ホストされたパスは重要です。終了パスも同様に重要です。ユーザーはいつでも、Raspberry Pi を含む低スペックのハードウェア上の自分のプロファイル ノードに移動できる必要があります。

それが利便性と保管性の違いです。

## Bitsocial Chain が強化する公開対話

Bitsocial Chain は、永続的な命名、決済、投げ銭、アワードなどの金融基盤を公開対話に直接組み込めます。

フェーズ3のクライアントは、投稿、テキスト、アイデアの共有、リアルタイムの議論を中心に据えながら、他の Bitsocial クライアントとコミュニティやネットワーク効果を共有します。
