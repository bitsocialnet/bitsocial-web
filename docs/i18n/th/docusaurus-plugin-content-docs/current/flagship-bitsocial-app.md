# กระจายศูนย์ Twitter/X

ระยะที่ 3 คือแผนสร้างทางเลือกแบบกระจายศูนย์ที่มุ่งเน้นแทน Twitter/X โดยมีการสนทนาสาธารณะที่ให้ความสำคัญกับข้อความเป็นศูนย์กลาง ได้แก่ โพสต์สั้น การตอบกลับ การรีโพสต์ การติดตาม การสนทนาแบบเรียลไทม์ และชุมชน พร้อมเปิดชั้นแพลตฟอร์มออกให้ใช้งานได้อย่างเสรี

Twitter/X ยังคงมีโพสต์ ข้อความ และการแบ่งปันแนวคิดเป็นแก่นหลัก ไคลเอนต์ระยะที่ 3 ควรแข่งขันด้วยประสบการณ์หลักนี้และทำให้มันทำงานได้อย่างยอดเยี่ยม

หน้านี้อธิบายทิศทางผลิตภัณฑ์ ไม่ใช่ข้อกำหนดการเปิดตัวที่ตายตัว อินเทอร์เฟซ ฟีดเริ่มต้น โมเดลโฆษณา ฟีเจอร์ AI และตลาด RPC ที่แน่นอนอาจเปลี่ยนได้เมื่อโปรโตคอลและแอปยุคแรกเติบโตขึ้น

## สิ่งที่ควรพิสูจน์.

ไคลเอนต์ควรพิสูจน์ว่าเครือข่ายสังคมแบบอิงโปรไฟล์สามารถหลีกเลี่ยงการกลายเป็นแพลตฟอร์มที่ผู้ให้บริการควบคุมข้อมูลประจำตัวและโปรไฟล์ของผู้ใช้:

- ผู้ใช้สามารถเป็นเจ้าของข้อมูลประจำตัวและโปรไฟล์ได้
- ชุมชนและโหนดโปรไฟล์สามารถเป็นแบบเพียร์ทูเพียร์ได้
- ชุมชนสามารถส่งเอฟเฟกต์เครือข่ายไปยังไคลเอนต์ Bitsocial ได้
- ผู้ให้บริการ RPC สามารถทำให้แอปสะดวกขึ้นได้โดยไม่ควบคุมข้อมูลประจำตัวหรือโปรไฟล์ของผู้ใช้
- อัลกอริธึมฟีดสามารถเป็นบริการเสริมแทนกฎหมายแพลตฟอร์มได้
- ไคลเอนต์อื่นยังสามารถแข่งขันในเครือข่ายเดียวกันได้

เป้าหมายคือการสร้างไคลเอนต์สำหรับการสนทนาสาธารณะที่ดีที่สุดเท่าที่ทำได้ และแสดงให้เห็นว่าโปรโตคอลสามารถขยายไปได้ไกลเพียงใด

## คุ้นเคยในจุดประสงค์ เปลี่ยนทดแทนได้ด้วยการออกแบบ

ประสบการณ์เริ่มต้นควรแข่งขันกับแก่นของ Twitter/X ได้ ได้แก่ ฟีดหน้าแรกที่รวดเร็ว โพสต์ข้อความ การติดตาม การตอบกลับ การกระจายผ่านรีโพสต์ ชุมชน การแจ้งเตือน การค้นหา และมุมมอง For You ที่จัดอันดับและใช้งานได้ทันที

Bitsocial Forge สามารถเรียกใช้ RPC และบริการฟีดเริ่มต้นแรกได้ ค่าเริ่มต้นดังกล่าวอาจรวมฟีดอันดับและโฆษณาเพื่อให้แอปรู้สึกเสร็จสมบูรณ์ตั้งแต่วันแรก แทนที่จะขอให้ผู้ใช้กระแสหลักประกอบสแต็กทั้งหมดด้วยตนเอง

ความแตกต่างคือค่าเริ่มต้นต้องไม่กลายเป็นคุก ผู้ใช้ควรสลับ RPC ฟีด อินสแตนซ์ ระบบจัดอันดับ โฆษณา และผู้ให้บริการค้นพบเนื้อหาได้ หรือปิดการจัดอันดับทั้งหมด ไคลเอนต์สามารถมีตัวเลือกเริ่มต้นที่ชัดเจนในการเปิดใช้ครั้งแรก โดยยังคงให้บริการหลักทุกอย่างเปลี่ยนทดแทนได้

สิ่งนี้ทำให้ไคลเอนต์ปรับแต่งได้มากกว่าแพลตฟอร์มทั่วไป ผู้ใช้คนหนึ่งอาจใช้ฟีดเริ่มต้นที่จัดอันดับและมีโฆษณาต่อไป อีกคนอาจใช้ฟีดตามลำดับเวลาโดยไม่มีการจัดอันดับ ส่วนอีกคนอาจเลือก RPC ที่เน้นความเป็นส่วนตัว บริการค้นพบที่ชุมชนดูแล ฟีดแบบชำระเงินที่ไม่มีโฆษณา หรืออัลกอริทึมเฉพาะกลุ่มสำหรับวัฒนธรรมย่อยหนึ่งๆ

## ชุมชนข้ามไคลเอนต์

ชุมชนควรมีความสำคัญมากกว่ากลุ่มที่แยกออกจากกันภายในแอปเดียว

บน X/Twitter ชุมชนถูกจำกัดอยู่ภายใน X ชุมชนเหล่านี้มีประโยชน์ แต่สืบทอดขีดจำกัดของแพลตฟอร์มเดียว ระบบบัญชีเดียว กลุ่มคำแนะนำหนึ่งรายการ และแพลตฟอร์มผลิตภัณฑ์เดียว

ชุมชน Bitsocial สามารถสร้าง โฮสต์ ค้นพบ และใช้งานผ่านไคลเอนต์ที่ต่างกันได้ นั่นหมายความว่าไคลเอนต์ระยะที่ 3 สามารถแสดงชุมชนและโพสต์จากเครือข่าย Bitsocial ที่กว้างกว่า ไม่ใช่เฉพาะจากผู้ใช้ที่เริ่มต้นภายในไคลเอนต์นั้น ชุมชนหนึ่งอาจมีกิจกรรมจากไคลเอนต์อิมเมจบอร์ด ไคลเอนต์สนทนาแบบ Reddit ไคลเอนต์ฟอรัมเฉพาะทาง แอปมือถือ และไคลเอนต์ระยะที่ 3 พร้อมกัน

นั่นคือข้อได้เปรียบหลักของเอฟเฟกต์เครือข่าย: ไคลเอนต์หนึ่งสามารถให้ความรู้สึกคุ้นเคยแก่ผู้ใช้กระแสหลัก ในขณะที่ยังดึงคุณค่าจากไคลเอนต์ โหนดชุมชน ผู้ให้บริการ RPC และบริการอิสระจำนวนมาก

## อัลกอริธึมฟีดเสริม

ไคลเอนต์ระยะที่ 3 ไม่ควรบังคับใช้ระบบจัดอันดับทั่วโลกเพียงระบบเดียวกับทุกคน

ควรเลือกใช้อัลกอริทึมฟีด ผู้ใช้สามารถเลือกอัลกอริธึมจากตลาดกลาง เปลี่ยนผู้ให้บริการ ใช้อัลกอริธึมจากบริษัท ใช้อัลกอริธึมที่ดำเนินการโดยผู้ดำเนินการที่ไม่ระบุชื่อ ใช้อัลกอริธึมที่สร้างโดยชุมชน เรียกใช้อัลกอริธึมส่วนบุคคล หรือไม่ใช้อัลกอริธึมเลย

ผู้ให้บริการ RPC สาธารณะเป็นสถานที่ที่บริการเหล่านี้สามารถแข่งขันได้ พวกเขาสามารถจัดทำดัชนี จัดอันดับ และแนะนำเนื้อหาได้ แต่ไม่ควรควบคุมข้อมูลประจำตัวหรือโปรไฟล์ของผู้ใช้

บริการเหล่านั้นยังสามารถแข่งขันกับรูปร่างของแอพได้ RPC หนึ่งรายการอาจมีฟีดจัดอันดับพร้อมโฆษณา อีกรายการหนึ่งอาจให้ฟีดตามลำดับเวลาที่ไม่จัดอันดับ อีกคนหนึ่งอาจเชี่ยวชาญด้านความเป็นส่วนตัว การแปล การกลั่นกรอง การค้นพบชุมชน หรือกราฟสังคมเฉพาะกลุ่ม

หากเศรษฐศาสตร์ใช้งานได้ บริการฟีดที่ได้รับการสนับสนุนจาก RPC สามารถเพิ่มฟีเจอร์ AI ที่คล้ายกับที่แพลตฟอร์มกระแสหลักพยายามใส่ลงในฟีดของตน: การแปลอัตโนมัติ สรุป การตอบกลับโดยใช้บอทช่วย คำตอบในการค้นหา ความช่วยเหลือในการกลั่นกรอง หรือบริบทสไตล์หมายเหตุของชุมชน

คุณลักษณะเหล่านั้นควรเป็นตัวเลือกบริการ ไม่ใช่ข้อกำหนดของโปรโตคอล RPC เริ่มต้นสามารถแข่งขันได้โดยเสนอฟีดที่สมบูรณ์ยิ่งขึ้น แต่ผู้ใช้และไคลเอนต์คู่แข่งควรจะสามารถเลือกทางเลือกที่เรียบง่าย เป็นส่วนตัว ตามลำดับเวลา ไม่มีโฆษณา หรือทางเลือกเฉพาะของชุมชนได้

## RPC ที่ไม่ควบคุมข้อมูลประจำตัวหรือโปรไฟล์ของผู้ใช้

ผู้ใช้แต่ละคนควรสามารถเข้าร่วมเป็นโหนดเพียร์ทูเพียร์แบบเต็มผ่าน RPC โดยไม่ต้องให้ผู้ให้บริการ RPC เป็นเจ้าของข้อมูลประจำตัวหรือโปรไฟล์ของตน

เส้นทางที่โฮสต์มีความสำคัญเนื่องจากผู้ใช้ส่วนใหญ่จะไม่เริ่มต้นด้วยการเรียกใช้เซิร์ฟเวอร์ เส้นทางทางออกมีความสำคัญไม่แพ้กัน: ผู้ใช้ควรจะสามารถย้ายไปยังโหนดโปรไฟล์ของตนเองบนฮาร์ดแวร์ที่มีสเปคต่ำ รวมถึง Raspberry Pi ได้ทุกเมื่อที่ต้องการ

นั่นคือความแตกต่างระหว่างความสะดวกสบายกับการยกสิทธิ์ควบคุมให้ผู้ให้บริการ

## การสนทนาสาธารณะที่เสริมพลังด้วย Bitsocial Chain

Bitsocial Chain สามารถนำการตั้งชื่อถาวร การชำระเงิน การให้ทิป รางวัล และกลไกทางการเงินอื่นๆ เข้าสู่การสนทนาสาธารณะได้โดยตรง

ไคลเอนต์ระยะที่ 3 ยังคงมีโพสต์ ข้อความ การแบ่งปันแนวคิด และการสนทนาแบบเรียลไทม์เป็นศูนย์กลาง พร้อมแบ่งปันชุมชนและเอฟเฟกต์เครือข่ายกับไคลเอนต์ Bitsocial อื่นๆ
